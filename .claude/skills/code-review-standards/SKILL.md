# コードレビュー基準スキル

実装完了後のレビューで用いる観点とチェックリスト。**レビューは実装とは別コンテキストで行う**（思い込みの持ち込みを避ける）。

## レビューの出力様式

各指摘は次を含める:
- **重大度**: `Blocker`(マージ不可) / `Major`(要修正) / `Minor`(改善) / `Nit`(任意)。
- **場所**: `path:line`。
- **根拠**: どの規約・原則に反するか（本スキルや `docs/` を参照）。
- **修正案**: 具体的なdiffや方針。

確信が持てない指摘は「要確認」と明示し、断定しない。良い点も簡潔に記す。

## 観点1: 契約と境界

- [ ] モジュール間がインターフェース(Protocol/interface)とドメインモデルで結合しているか。具体実装に直接依存していないか。
- [ ] 依存方向が一方向か（domain は外向き依存ゼロ）。
- [ ] 境界の入出力が Pydantic/Zod 等で検証されているか。
- [ ] ドメインモデルがフレームワーク(ORM/HTTP)に汚染されていないか。

## 観点2: 型と宣言性

- [ ] 型ヒント/型注釈が網羅され、`Any`/`object`/`as any` を濫用していないか。
- [ ] 新しい型記法（py: `dict`/`list`/`X | None`、TS: 厳密な型）を使っているか。
- [ ] 命令的手続きより宣言的に書けないか（早期return・純粋関数・不変データ）。

## 観点3: 可観測性（重点）

`observability-logging` スキルと `docs/observability/` を基準に:
- [ ] 構造化ログか（`print`/素の`console.log`がないか）。
- [ ] アプリログに `trace_id`/`span_id` が相関しているか。
- [ ] メッセージに可変値を埋め込んでいないか（固定`body`＋属性）。
- [ ] 重大度が適切か（4xx=WARN、5xx/未捕捉=ERROR、ERROR以上に`exception.*`）。
- [ ] 機密がマスクされるか（新規フィールドはマスク対象＋テスト追加）。
- [ ] 認証認可イベントが監査ログ(`event.domain`)として分離されているか。
- [ ] AI呼び出しが `gen_ai.*` 計装を通り、プロンプト本文が既定で出ないか。

## 観点4: セキュリティ

- [ ] 認可判定があり、拒否が監査記録されるか。
- [ ] 秘密/トークンがログ・コード・コミットに混入していないか。
- [ ] 入力検証・SQLインジェクション(パラメータ化)・CORS/CSRF設定。
- [ ] JWT検証（署名/exp/aud/iss）が正しいか。

## 観点5: テスト（SSOT）

- [ ] 契約・境界がテストで固定されているか（詳細設計書に頼っていないか）。
- [ ] ドメイン/ユースケースが実装詳細でなく振る舞いをテストしているか。
- [ ] 可観測性規約のテスト（相関/マスキング/重大度/監査）があるか。
- [ ] 失敗系・境界値・例外パスを網羅しているか。

## 観点6: 保守性・依存

- [ ] 依存追加が正当か（過度に増やしていないか）。
- [ ] 命名・配置がレイヤ構成と一致するか。
- [ ] ドキュメンテーションコメント（日本語・規約準拠）が要点に付いているか。
- [ ] 重複が共通化されているか（ただし早すぎる抽象化に注意）。

## 進め方

1. 変更範囲を把握（diff/対象ファイル）。
2. 上記観点で走査し、重大度順に指摘を列挙。
3. Blocker/Major を優先して修正提案。
4. 修正後、観点3(可観測性)とテストの再確認。
