# FlowNote — コンセプトと提供価値

> 本書は題材システム「FlowNote」のコンセプト・提供価値・スコープを明文化する。
> **重要**: 本プロジェクトの主目的は[可観測性/ログシステムの構築と観察](./observability/observability-architecture.md)であり、FlowNote はその器である。機能の作り込みより、各レイヤ（フロント/バック/AI/認証）でログがどう設計・相関・収集されるかの観察を優先する。

## 1. 一行コンセプト

**「Markdownで書いたメモが、AIの助けでタスクと知識に育ち、変更履歴で守られる個人ワークスペース」**

## 2. 解決する課題

知的作業者は、断片的なメモ・やるべきこと・過去の経緯が**別々のツールに散在**し、文脈を失いがちである。FlowNote は次を1つに統合する:

- **メモ(Notes)**: Markdownファイルベースの記述。
- **タスク(Tasks)**: メモから派生し、進捗を管理。
- **バージョン管理(Versions)**: メモの変更履歴・差分・復元。
- **AI**: 進捗の相談、内容の整理、横断検索を支援。

## 3. 提供価値（Value Proposition）

| 利用者の課題 | FlowNote の価値 |
|---|---|
| メモがタスクに繋がらない | メモから1クリックでタスク化。AIが粒度を提案。 |
| 過去の記述が失われる/上書きされる | すべての保存をバージョンとして記録し、差分・復元が可能。 |
| 「何から手を付けるか」が分からない | AIが進捗をレビューし、次の一手を相談できる(`progress_review`/`task_consult`)。 |
| 必要な情報が見つからない | メモ・タスク・履歴を横断する**AI統合検索**(`unified_search`)。 |
| ツールが散在する | 1つのワークスペースに統合。 |

### プロジェクトとしての提供価値（本来の主目的）

- **観察可能な実例**: AI・認証認可を含む現実的構成で、3本柱(ログ/トレース/メトリクス)が相関する様子をローカルで観察できる。
- **再利用可能な規約とスキル**: [ログ規約](./observability/logging-spec.md)と `.claude/skills/` のスキル群は、他プロジェクトへ流用できる。

## 4. 想定ユーザーとロール

| ペルソナ | 説明 | ロール([RBAC](./observability/audit-logging.md)) |
|---|---|---|
| 閲覧者 | 共有された内容を見る | `viewer` |
| 作業者 | 日常的にメモ/タスクを編集しAIを使う | `editor` |
| 管理者 | ユーザー管理・削除・監査閲覧 | `admin` |

## 5. 主要ユースケース（題材として「一通り」実装）

1. **メモCRUD**: Markdownメモの作成・閲覧・更新・削除。
2. **バージョン管理**: 保存ごとにバージョン生成、履歴一覧、差分表示、特定版へ復元。
3. **タスクCRUD**: タスクの作成・状態遷移(`todo`/`doing`/`done`)・メモとの紐付け。
4. **AI相談(`task_consult`)**: タスク/メモを文脈に、進め方を相談。
5. **AI進捗レビュー(`progress_review`)**: 滞留タスクの抽出と次の一手の提案。
6. **AI統合検索(`unified_search`)**: メモ/タスク/履歴を横断した意味検索。
7. **認証認可**: Keycloak(OIDC)ログイン、ロールに応じた操作可否。

### 垂直導線（ログ観察の主動線）

> 「メモ作成 → AI統合検索で関連を発見 → タスク化 → AIに進捗相談」

この1本の導線がフロント→API→ドメイン→DB→AI→認証の**全ログ面**を貫く。観察の主対象とする。

## 6. スコープと非スコープ

**スコープ（実装する）**
- 上記7ユースケースの主要機能。
- 全レイヤを貫く可観測性（本プロジェクトの本体）。
- ローカルで動く構成（docker-compose + ローカル開発コマンド、AIはモック）。

**非スコープ（作り込まない）**
- リッチなエディタUX、リアルタイム共同編集、モバイル対応。
- 本番運用のスケーリング/HA、課金。
- AIの高度なRAG基盤（検索は素朴な実装＋計装の観察が目的）。

## 7. 品質の優先順位

1. **可観測性の規約準拠**（最優先・テストで固定）。
2. 契約と境界の明確さ（インターフェース＋ドメインモデル、[アーキテクチャ](./architecture.md)）。
3. 型安全・宣言的・テスト網羅。
4. 機能の網羅性（最後）。

## 8. 成功の定義

- 1つの `trace_id` で、ブラウザ操作からAI/DBまでを横断追跡できる。
- ログが[規約](./observability/logging-spec.md)に準拠し、機密がマスクされ、監査/業務ログが分離されている。
- 規約がテストで固定され、`uv run pytest` / `pnpm vitest` が緑。
- 規約・スキルが他プロジェクトに転用できる形で残る。
