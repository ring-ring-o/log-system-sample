# ADR 0006: `service.instance.id` の運用方針（環境別の採番元）

- ステータス: 採用
- 日付: 2026-06-01

## 背景

[アーキテクチャ §6](../observability/observability-architecture.md) で `service.instance.id` をリソース属性に挙げているが、**何を入れるか**が未定義だった。これは「どのプロセス/レプリカが出したシグナルか」を識別する軸で、スケールアウト時の障害局所化やローリングデプロイの観察に必要。

## 決定

`service.instance.id` の値は**環境（デプロイ形態）ごとに最も安定して一意な識別子**を採用し、`OTEL_SERVICE_INSTANCE_ID`（または OTel リソース検出）で注入する。アプリのコードに採番ロジックを持たない（合成ルートで環境変数から読むだけ）。

| 環境 | 採番元 |
|---|---|
| `local` | プロセス起動ごとの UUID（観察できれば十分） |
| Kubernetes | **Pod 名**（`metadata.name` を downward API で注入）。再起動で変わるが Pod 単位の識別に最適 |
| その他コンテナ基盤 | リビジョン/レプリカのインスタンスID（基盤が払い出す値） |
| VM/ベアメタル | ホスト名 + プロセス起動 UUID |

## 理由

- OTel 推奨は UUID（v1）だが、**運用で人が突き合わせる**には Pod 名等の基盤ネイティブ識別子の方が有用（ログ/メトリクス/ダッシュボードと往復しやすい）。
- 採番を基盤（downward API/リソース検出）に委ねることで、アプリは環境非依存を保てる。

## 留意 / リスク

- Pod 名は再起動で変わるため、**長期の同一性**は `service.name`+`service.version`+（デプロイ識別）で担保し、`service.instance.id` は「いまどのレプリカか」に用いる。
- 高カーディナリティになりうるため、**メトリクスのラベルには載せない**（リソース属性に留める）。[logging-spec §4.3](../observability/logging-spec.md) のカーディナリティ規約に従う。

## 代替案

- 常に UUID 固定 → 人手の突き合わせがしづらい。基盤識別子を優先。
